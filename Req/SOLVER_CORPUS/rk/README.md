# RK — Khovratovich's original Equihash reference, C++ and Rust

Two implementations of Dmitry Khovratovich's original CC0-licensed 2016
Equihash reference (`equihash-khovratovich/Source/C++11/pow.h`/`pow.cc`,
[github.com/khovratovich/equihash](https://github.com/khovratovich/equihash)):
the untouched original in `original/`, and a faithful line-for-line Rust
port in `src/`. This predates the entire 2016-17 optimization wave
(`../../Equihash.md` §2/§3) — no index-pointer compression, no
incomplete bucket sort, no arena reuse — so it is close to a full-index
upper bound on memory, not a tuned solver.

The `Equihash` class's five methods are preserved with their original
names and control flow in the Rust port: `initialize_memory`
(`InitializeMemory`), `fill_memory` (`FillMemory`), `resolve_collisions`
(`ResolveCollisions`), `find_proof` (`FindProof`), and
`resolve_tree`/`resolve_tree_by_level`
(`ResolveTree`/`ResolveTreeByLevel`). Neither implementation has any
CLI/console dependency in its own algorithm module — each has separate
driver binaries for I/O.

## Parameter range and validity

`n` up to 256 bits, `k` bounded by `n/(k+1) <= 32` (the original's own
documented `MAX_N=32`-byte assumption) — broader than RT's
compile-time-restricted range, since this is a generic recursive
tree-fold over dynamically-sized containers, not bucket-specialized C.
Reaching `(192,7)`/`(200,9)` needs no port change, only running at those
parameters (not attempted — see "Open questions" below).

Both drivers below reject an invalid `(n,k)` immediately, naming which
rule failed, before any solve work starts:
1. `k >= 1`
2. `n % (k+1) == 0` (algorithm-fundamental: `ell=n/(k+1)` must be a
   whole number)
3. `n/(k+1) <= 32` (the original's own `MAX_N` assumption)

Note this is genuinely NOT the same rule set as Req/Equihash's or
CS/Sequihash's own — RK's `hash_leaf` always hashes to a fixed 32-byte
digest and extracts bits via runtime shift, so it needs no byte
alignment (`n%8==0` does not apply here, unlike Req/CS).

## Build environment — the original does not build on arm64 unmodified

The upstream Makefile (`-maes -mavx`) and bundled `blake/blake2.h`/
`blake2b.cpp` are x86-only (AVX/SSE2 intrinsics, plus a packed-struct
ABI bug unrelated to that). This port keeps `pow.h`/`pow.cc` untouched
except `rdtsc()` (a portable `std::chrono` fallback, algorithmically
inert — only affects printf timing output); the bundled BLAKE2b header
is replaced with a 20-line stub declaring only `blake2b`, linked against
the repository's vendored `BLAKE/vendor/blake2/blake2b-ref.c` (Samuel
Neves, CC0, portable, no intrinsics) through a small glue translation
unit matching `pow.cc`'s exact expected call signature — portable, so
the same build works unchanged on x86_64.

`original/pow.h` has one further addition beyond the upstream source: a
public `Equihash::SetNonce(Nonce)` setter (needed for the single-attempt
benchmark driver below; `nonce` has no public accessor and isn't
reachable via `#define private public` either, since it has no explicit
`private:` keyword). No existing method's behavior changed.

## Validation

8 KAT vectors (`vectors/*.json`), generated by running the original C++
once and discarded — not a live subprocess dependency. Each vector
records `{n,k,seed,nonce,indices,verified}`; `tests/cross_check.rs`
re-runs the Rust port's own `find_proof()` (replicating the original's
multi-nonce retry search exactly) and asserts both the returned nonce
and full index array match. All 8 pass.

```
cargo test --release   # debug build takes ~3 min due to (126,5)'s search cost
```

The original defines no minimal/compressed wire encoding — raw index
arrays only. This port makes no wire-format claim.

## Current benchmark numbers (single-attempt, fixed nonce, both languages)

Both languages measure the SAME quantity: one
`initialize_memory`+`fill_memory`+`resolve_collisions×k` pass at a
caller-supplied nonce, not `find_proof`'s multi-nonce retry search
(which would make Rust's allocator-peak and C++'s OS-RSS answer
different questions — see `rk_bench` vs. `rk_single_gen`/
`rk_single_bench` in "Layout" below). Rust's figure is allocator bytes
requested; C++'s is OS-reported resident set size — related but not
identical quantities (RSS runs somewhat above requested bytes from
allocator bookkeeping); a ratio near 1.0x should not be read as the
literal same bytes.

5-rep min-of-reps wall time, this machine (Apple M4 Pro, arm64, 48GB
RAM), seed=5, nonce=2:

| (n,k) | Rust wall | C++ wall | wall ratio | Rust mem | C++ RSS | mem ratio |
|---|---|---|---|---|---|---|
| (60,4) | 2.9ms | 2.3ms | 1.27× | 2.2 MB | 4.3 MB | 0.52× |
| (70,4) | 11.8ms | 9.8ms | 1.20× | 9.0 MB | 12.4 MB | 0.73× |
| (80,4) | 61.4ms | 52.3ms | 1.17× | 35.9 MB | 44.8 MB | 0.80× |
| (90,5) | 29.2ms | 25.4ms | 1.15× | 19.3 MB | 29.3 MB | 0.66× |
| (96,5) | 74.4ms | 57.2ms | 1.30× | 38.5 MB | 71.3 MB | 0.54× |
| (100,4) | 1699.7ms | 1419.5ms | 1.20× | 574.6 MB | 693.6 MB | 0.83× |
| (102,5) | 184.5ms | 157.3ms | 1.17× | 77.1 MB | 113.6 MB | 0.68× |
| (108,5) | 428.5ms | 344.3ms | 1.24× | 154.1 MB | 224.3 MB | 0.69× |
| (110,4) | 7062.5ms | 5877.9ms | 1.20× | 2298.5 MB | 2769.3 MB | 0.83× |
| (114,5) | 952.2ms | 752.0ms | 1.27× | 308.3 MB | 448.7 MB | 0.69× |
| (120,4) | 33.5s | 29.3s | 1.15× | 9.19 GB | 11.07 GB | 0.83× |

Stable pattern across every point, no crossover: **Rust is consistently
1.15–1.30× C++'s wall time and consistently 0.52–0.83× C++'s memory** —
comfortably inside a 2×/1.5× bar, and on memory, better than C++, not
worse. `(120,4)` is the practical ceiling on this machine (Rust just
under, C++ just over, a 10 GB budget); wall time (~30s) is nowhere near
a 200s budget — memory is the binding constraint for this solver's k=4
family, not time.

Reproduce: `cargo build --release --bin rk_single_gen` and
`cmake --build original/build`, then run both at the same
`(n,k,seed,nonce)`. Each invocation writes a new file to that
implementation's own `runs/<n><k>_<timestamp>.jsonl`.

## Layout

- `src/lib.rs` — the ported algorithm. Entry points: `Equihash::new`,
  `.find_proof()`, `.single_attempt(nonce)`, `Proof::test()`,
  `check_nk(n,k)`.
- `src/bin/rk_gen.rs` — `rk_gen <n> <k> <seed>`, one correctness JSON line.
- `src/bin/rk_single_gen.rs` — `rk_single_gen <n> <k> <seed> <nonce>
  [--reps N] [--no-save]`, the benchmark driver above.
- `src/bin/rk_bench.rs` — full-`find_proof` benchmark (multi-nonce
  retry included in the timing; superseded by `rk_single_gen` for
  cross-language memory comparison specifically, still useful for
  realistic end-to-end timing).
- `tests/cross_check.rs` — validates against `vectors/`.
- `original/` — the untouched upstream source plus this port's own
  `rk_single_bench.cpp` (the C++ benchmark driver) and `rk_vecgen.cc`
  (vector regeneration).
